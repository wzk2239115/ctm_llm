## 测试内容

本阶段测试增加 CTM tick 数是否能产生有用的更深层计算：

- 固定 tick 数从 1 到 16；
- tick8 下不同的 tick 损失模式；
- tick8 和 tick16 下的停机模式；
- 停机阈值和置信度温度。

## 为什么需要此实验

Tick 是 CTM 的核心机制。如果 tick 深度有效，模型应至少表现以下之一：

- 损失随 tick 数增加而降低；
- 出现自适应的有效 tick 行为；
- 通过停机以更低计算维持相似质量；
- 后续 tick 中有明显改善。

## 结果

最佳固定 tick 结果为 tick2：

| Tick | 损失 | Tok/s | 峰值显存 MB |
| ---: | ---: | ---: | ---: |
| 1 | 5.4104 | 6921 | 18990 |
| 2 | 5.3994 | 4214 | 29813 |
| 3 | 5.4598 | 3085 | 40397 |
| 4 | 5.4903 | 2403 | 50989 |
| 6 | 5.5884 | 1704 | 60800 |
| 8 | 5.5597 | 1159 | 63195 |
| 12 | 5.7765 | 634 | 47515 |
| 16 | 5.7766 | 442 | 61801 |

Tick8 损失变体未改变结论：

- `min_conf`：损失 `5.5529`；
- 基础 tick8：损失 `5.5597`；
- mean：损失 `5.5728`；
- last：损失 `5.5789`。

停机变体显示了有效 tick 值，但未能充分改善损失。tick16 阈值停机通常报告有效 tick `16.0`，意味着并未真正节省计算。tick16 附近的置信度停机报告有效 tick 约 `8.48`，但损失仍在 `5.78` 至 `5.81` 左右。

## 结论

更多 tick 当前并未产生更好的推理。有用区间为 tick1 至 tick2。超出此范围，模型付出大量计算和显存代价，质量反而下降。

## 优势或劣势诊断

劣势的可能原因：

- Tick 监督未迫使后续 tick 添加有用信息。
- 重复 tick 可能过度处理浅层 token 表示。
- 随 tick 数增长优化变得更困难。
- 显存因 tick trace 和逐 tick 输出而增长。
- 停机标准尚未与实际有用计算对齐。

可能的优势：

- Tick2 相比 tick1 略有改善，说明少量循环计算是有帮助的。

## 后续

1. 将 tick2 设为下一轮的默认研究设置。
2. 增加 tick 课程学习：
   - 先训练 tick1/tick2；
   - 逐步引入 tick4+；
   - 正则化后期 tick 使其优于早期 tick。
3. 重新设计停机：
   - 停机应减少实际计算，而非仅产出有效 tick 指标；
   - 在相同平均计算量下对比固定 tick2 与动态 tick。
4. 增加逐 token tick 分析：
   - 简单 token 应早停；
   - 困难 token 应使用更多 tick；
   - 报告 token 级别有效 tick 直方图。
